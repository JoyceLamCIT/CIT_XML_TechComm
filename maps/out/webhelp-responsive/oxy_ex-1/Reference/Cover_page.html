Cover Page

This is the cover page with general information about the techniques, elements, and how I've used them in this project.

Conditional processing and flagging schemes

Created the IBM_StoredIQ_filter.ditaval file to filter for general and expert user and exclude novice users. For platforms, it includes Windows, excludes Mac, and flags Linux with absurd colors.

Chunking

Walls of text or unmanageable lists were broken down into sections, tables, or topics according to concept, reference, or task information types.

indexterm element

Used a combination of indexterm, index-see, and index-see-also to create an index of
  • 8 regular entries
  • 20 nested entries
  • 7 synonymous entries

href element

If the source material linked to other URLs outside the document, they were included in the DITA as href.

If the source material linked to another spot in the document that was not transformed for Project 2, these also became href linking back to the source PDF.

xref element

If the source material linked to another spot in the document that was transformed, xref was used in the DITA. This included linking to other topics, sections, and individual steps.

If the source material linked to another spot in the document that was not transformed for Project 2, these became href linking back to the source PDF.

conkeyref attribute

Used for inserting image content.

reltable element

Linked topics of relevance together.

related-links element

I used related links to both topics as well as elements within topics. In container topics, I used related-links and linklist to create a mini TOC.

audience attribute

I believe the audience to be a general administrator tasked with installing the IBM StoredIQ. This information is in the metadata.

platform attribute

I made the assumption that the platform was on Windows, and inserted this into the metadata.

product attribute

The product is the IBM StoredIQ, and it was inserted into the metadata.

note element

Notes had type and importance attributes set according to their gravity and purpose identified in the source content.

example element

I used this element to wrap around examples in the source content.

prolog element

I included copyright information, other product information, and created dates for versioning purposes. I assume for later iterations, the "modified" attribute will be added to the dates.

cmdname element

Used to tag command names in the software.

substep element

Used to break down steps into series of actions.

stepresult element

Used to provide the outcome of a step so the user understands what is to be expected.

info element

Used inside steps to provide additional information.

postreq element

Ysed to describe the steps or tasks that the user should do after the successful completion of the current task.

prereq element

Used to inform the user about the the things the user needs to know or do before starting the current task.

userinput element

Used for anything I interpretted as elements the user can change or input, such as usernames, passwords, and text values.

uicontrol element

Used for anything I interpretted as interactive elements within the user interface.

menucascade element

Used for multiple uicontrol elements in sequential order.

codeph element

Used to wrap snippets of program code inline with the content.

codeblock element

Used to wrap blocks of program code.

glossgroup element

I created a glossgroup topic to demonstrate that I understand how to group multiple glossentry elements into one topic.

glossentry element

I created glossentry topics and mapped them into the project map to show I understand they can be mapped as well. I understand they can be mapped into a glossary.ditamap, but I did not want to have 2 glossaries since I also created a glossgroup.

topichead element

Used to create letter sections for the glossary.

abbreviated-form element

Used to call out instances of glossary terms.

parml element

Used parml, plentry, pt, and pd to list terms and definitions described as parameters.
